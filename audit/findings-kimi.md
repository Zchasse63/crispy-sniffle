  ## Executive summary

  **Verdict: Not launch-ready for Miami scale.** The codebase is genuinely disciplined — provenance tagging is honest, prompt-injection containment is strong, admin authz is correctly gated everywhere I checked, owner-invite tokens are hashed/single-use, the auth callback's `?next` guard is solid, and the dreaded enum drift is currently *clean*. But Tampa's 36→747 growth left structural scale bugs that Miami (~1,800) will turn into outages, and there is one exploitable stored-XSS hole. The test suite cannot catch any of this: there is no CI, and the e2e suite runs against live production data.

  The 10 biggest risks:

  1. **P0** — `gyms.website` flows scraped/owner/admin input verbatim into `<a href>` with zero scheme validation at any write path; CSP `script-src 'unsafe-inline'` lets a `javascript:` payload execute and steal Supabase sessions (staff included).
  2. **P1** — Every gym detail page view re-fetches the *entire city catalog* (`fetchCityGyms` + ~36-chunk × 4-table fan-out) just for 4 "similar spots" cards and price bands — at Miami that's ~145 PostgREST requests per page view.
  3. **P1** — City pages pay the full catalog fetch **twice** per view (`generateMetadata` + render, no `cache()` dedupe).
  4. **P1** — `sitemap.ts` has no `.range()` pagination: gyms alphabetically past row 1,000 silently vanish from the sitemap exactly as Miami lands.
  5. **P1** — Public-read RLS is `using (true)` on every catalog table; the `is_live`/`status` gates exist only app-side, so the shipped anon key reads non-live metros, closed gyms, and internal `status_note`s.
  6. **P1** — 9 `AmenityKey`s (e.g. `chalk_allowed`) are absent from the `amenities` catalog table that `gym_amenities.amenity_key` FK-references — an owner checking one of them makes the entire submission publish fail on an FK violation.
  7. **P1** — `GymCard` hardcodes `neighborhood ?? "Tampa"` and has no photo `onError` fallback: Miami's grid will show wrong-city labels and broken-image icons on unrehosted scraped photos.
  8. **P1** — Both LLM edge functions derive the rate-limit IP from the spoofable leftmost `x-forwarded-for`; per-IP limits are bypassable and only the global daily cap bounds abuse.
  9. **P1** — The fallback NL parser matches synonyms by raw substring: "boxing" contains "box" → phantom CrossFit equipment requirements; "indoor track" → `squat_rack`; "under 30 minutes" → `maxDayPass=30`.
  10. **P1** — No CI, no `test:e2e` script, e2e asserts against live prod data — the two scale bugs that already shipped were structurally uncatchable, and one instance (sitemap) is still live.
